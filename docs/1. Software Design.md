# Software Design

## Layering

| Layer | Path |
|-------|------|
| HTTP | `src/http/router.rs`, `src/http/dto.rs` |
| gRPC | `src/grpc/mod.rs` |
| Application | `src/service/wallet_service.rs`, `src/service/reconciliation.rs` |
| Domain | `src/wallet.rs` |
| Persistence | `src/persistence/repositories.rs`, `models.rs` |
| Payments | `src/payment/mod.rs` (`MidtransGateway`) |
| Messaging | `src/messaging/provisioning_consumer.rs` |
| Composition | `src/server.rs`, `src/main.rs` |

## SOLID

- **SRP:** `Wallet` domain types vs `WalletService` orchestration vs `MidtransGateway` payment I/O.
- **OCP:** New transaction types extend `wallet.rs` enums without rewriting all HTTP handlers.
- **LSP:** Repository traits allow test doubles in `tests/service_tests.rs`.
- **ISP:** gRPC surface limited to hold/release/convert operations auction needs.
- **DIP:** `WalletService` uses repositories and `MidtransGateway`, not raw SQL in handlers.

## Patterns (≥3)

### 1. Repository pattern

`WalletRepository`, `TransactionRepository`, `ProvisioningEventRepository` in `persistence/repositories.rs`.

### 2. Facade pattern

`MidtransGateway` in `payment/mod.rs` hides Core API + IRIS disbursement complexity.

### 3. Service layer

`WalletService` coordinates domain + persistence + external gateway (documented in source comments).

### 4. Message consumer (event-driven)

`messaging/provisioning_consumer.rs` creates wallets on `WalletProvisionRequested` from auth.

### 5. Adapter (HTTP + gRPC)

Same business rules exposed via `http/router.rs` and `grpc/mod.rs`.
